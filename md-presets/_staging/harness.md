# HIZZI Board — 하네스

> 공장(Claude Code)이 일을 수행할 때 참조하는 매뉴얼. 실행·검증의 단일 출처.
> 세션 경계 절차(시작·종료·프리셋)는 session.md 참조.
> 수정: 오너 승인 필수.

## 0. 경계 — 기획과 공장

기획 대화(오너 ↔ Claude.ai)는 공장 밖, 실행·검증·보고(Claude Code + Codex)는 공장 안이다. 경계의 내용: 무엇을·왜는 기획(공장 밖), 어떻게는 공장(안). 변수 선언·함수 시그니처·hook 호출 순서·API 호출 패턴·store 구문 같은 구현 구조는 공장의 1-1 탐색·1-2 구현에서 결정한다. 공장 밖에서 미리 박지 않는다.

공장 안엔 Code와 Codex가 각자 책임으로 공존한다. Code는 프롬프트를 현실(파일·빌드·실행)로 만드는 집행자, Codex는 그 결과물이 실효적으로 맞는지 주관적으로 검증하는 검열자. 서로 대체할 수 없는 역할 분리다. "공장에 넘긴다"고 할 때 둘을 한 덩어리로 보지 않는다. 공장장(Code)은 주문을 받아 혼자 돌되 검증 단계마다 Codex를 호출하고, 공장 밖은 공정에 개입하지 않는다. 오너의 복붙은 한 번 — 주문 한 번. 보고는 공장이 progress.md에 직접 쓴다.

```
오너 ↔ Claude.ai (기획 대화)
      ↓ 일반어 명령 한 번 복붙

공장 (Claude Code + Codex)
├─ 1-1 탐색 ───── 저장소(grep) + 오픈소스
│       ↓
├─ 1-2 구현 ───── 코드 작성 (Code)
│       ↓
├─ 1-3 빌드 검증 ─ 파일 존재 / 빌드 통과 (Code)
│       ↓ 실패 → 구현 복귀
├─ 1-4 코드 리뷰 ─ Codex (가동/기능/디자인 3축)
│       ↓ 실패 → 구현 복귀 또는 Codex rescue
├─ 1-5 E2E ──── Playwright (조건 해당 시 필수)
│       ↓ 실패 → 구현 복귀
└─ 1-6 배포+보고 ─ vercel 배포 + progress.md 직접 업데이트

막힘 탈출구 — 같은 실패 2회 / 설계 판단 필요
        → Claude.ai 재질의 (ask-claude.js)
        → 답 받고 공장 복귀
```

## 1. 공장 6단계

작업 크기와 무관하게 6단계를 모두 거친다. 단순한 코드도 쌓이면 단순하지 않고, 예외 구멍은 관습 회귀의 진입점이 된다. 예외는 1-5 E2E 하나뿐 — 트리거 조건에 해당하지 않는 변경만 건너뛴다(해당되면 필수 통과).

### 1-1. 탐색

**저장소 내부** — grep 기반 경량 탐색을 기본으로 한다. `@explorer` 서브에이전트는 과거 환각으로 자료를 준비한 케이스가 관측돼 신뢰도 이슈가 있다. grep으로 관련 파일·패턴을 실제 텍스트 매칭으로 찾고, 필요하면 `view` 도구로 해당 파일을 직접 읽어 구조·명명·스타일을 파악한다. 기존 코드에 이식되지 않고 자연스럽게 녹는 것이 목적.

**오픈소스** — 새 기능이나 범용 도메인 문제는 외부 라이브러리 조사를 먼저 한다 (R4.9). 범용 도메인 = 날짜·시간·캘린더·마크다운·드래그앤드롭처럼 히찌보드 밖에서 표준화된 문제. 검증된 라이브러리가 있으면 자체 구현보다 낫다.

단순 수정이어도 탐색은 생략하지 않는다. grep 기반이라 비용이 낮고, 단순해 보이는 변경이 기존 코드와 어긋나는 일이 반복됐다.

### 1-2. 구현

`@implementor`가 탐색 요약을 받아 코드를 쓴다. 소규모 수정은 공장장 본체가 직접 쓰기도 한다.

**선택 단계 — 설계 검토 (`/codex:adversarial-review`)**
큰 구조 변경·설계 판단이 걸리면 구현 전에 Codex에게 "이 설계로 가는 게 맞나? 놓친 실패 모드는?"을 물을 수 있다. 단순 작업엔 불필요.

### 1-3. 빌드 검증 (필수)

`@implementor`의 완료 보고를 믿지 않고 실제 파일과 빌드를 확인한다. 2026.04.18 테스트에서 "완료" 보고 후 파일 쓰기가 안 된 trust-but-verify 이슈가 관측됐기 때문.

```bash
ls -la <변경 파일 경로> && npm run build 2>&1 | tail -3
```

- 파일 미존재 → Write 도구로 공장장 본체가 재생산
- 빌드 실패 → 구현 단계로 복귀

**건너뛸 수 없는 단계다.** 빌드 검증 없이 넘어간 적이 있다면 공장이 고장난 상태.

### 1-4. 코드 리뷰 — `/codex:review` (필수)

빌드 검증을 통과한 코드를 Codex가 리뷰한다. 세 축으로 본다.

- **가동 축** — 빌드·타입·런타임 오류 없이 실제로 도는가
- **기능 축** — `rules.md` / `flows.md` 기준 충족, 놓친 엣지케이스, 기존 코드와의 일관성
- **디자인 축** — 색상 토큰 `uxui.md` 준수 / transition 0.15s ease 통일 / z-index 계층 이탈 없음 / 공통 컴포넌트 인라인 재구현 없음

3축 모두 충족해야 PASS. 1축이라도 생략되면 "미검증" 플래그를 달고 다음 단계 진입 금지 (R4.10).

결과 처리:
- **PASS** → 다음 단계
- **경미한 수정** → 구현 복귀
- **공장장이 고치기 어려운 문제** → `/codex:rescue` (Codex가 직접 수정·리뷰 통과까지 책임)

`/codex:review`가 Codex 플러그인 빌드에 따라 노출되지 않을 수 있다. 그 경우 `/codex:rescue`를 "read-only 리뷰 모드"로 지정해 대체 사용.

**건너뛸 수 없는 단계다.** 작업 크기와 무관.

### 1-5. E2E — Playwright (조건 해당 시 필수)

브라우저 동작(클릭·입력·비동기 상태·레이아웃)에 영향을 주는 변경에 트리거된다. 트리거 조건에 해당하면 **건너뛸 수 없다** — 타입 체크·빌드·코드 리뷰가 이미 통과했어도 브라우저에서만 드러나는 버그는 E2E가 잡는다.

**자동 트리거 조건:**
- PR 라벨 `needs-e2e`
- `src/components/**Modal*.tsx` 변경
- `src/components/calendar/**` 변경
- `src/components/**Form*.tsx` 변경
- `src/components/**Button*.tsx` 변경
- `src/app/**/page.tsx` 변경

**수동 트리거:** Claude Code 터미널에서 Playwright MCP 직접 호출.

트리거 조건 밖의 변경(유틸 함수, 타입 정의, 서버 로직, 스타일 토큰 등 브라우저 동작에 영향 없는 변경)은 E2E 생략. 이게 공정에서 유일하게 허용된 예외다.

현재 spec: `playwright-login.spec.js` (인증 불필요). 실패 시 구현 복귀.

**결과물 저장 경로:**

Playwright 실행 결과물은 아래 경로에 저장하고 `.gitignore`로 git 추적 제외한다.

- `test-results/` — 스크린샷·trace·video (실패 분석용)
- `playwright-report/` — HTML 리포트

**실패 시 보고 필수 항목:**

E2E 실패 시 Code는 구현 복귀 전에 아래를 수집해 보고한다:
- 실패한 테스트 케이스 이름
- 실패 시점 스크린샷 경로
- trace 파일 경로 (원인 추적이 필요한 경우)
- 콘솔·네트워크 로그 중 관련 부분 발췌

막힘 탈출구로 Claude.ai에 질의할 때도 위 자료 경로를 ask-claude.js 메시지에 포함한다.

**보존 정책:**

성공 시 결과물 즉시 삭제 (Playwright 기본 동작). 실패한 테스트만 유지해 디버깅·재점검에 사용.

### 1-6. 배포 + 보고 (필수)

작업 1건의 완결은 "코드 머지"가 아니라 "사용자가 실제로 쓸 수 있는 상태"다. 1-6은 배포와 보고를 한 공정으로 묶어 처리한다.

**배포:**
`npx vercel --prod` 실행. 배포 성공 시 다음 단계. **실패 시 멈추고 실패 원문 그대로 보고.** 재시도나 복귀 없음 — 인프라 이슈는 오너 진단 영역.

**보고 (배포 성공 후):**
공장장이 progress.md를 직접 업데이트한다. 오너에게 텍스트를 넘기는 방식이 아니라 파일을 바로 쓰고 커밋.

- **현재상태 섹션 갱신** — 완료된 TODO 제거, 필요 시 후속 TODO 추가, 다음 1순위 조정
- **작업로그 섹션 append** — 방금 끝난 작업을 짧은 한 줄로 기록: `- [YYYY-MM-DD] 작업명 (커밋 해시) — 영향 파일`

세션 중 공장이 여러 번 돌면 1-6도 여러 번 발생한다. 매 회 위 두 섹션에 반영돼 언제든 progress.md를 열면 최신 상태다. 세션 단위 요약 블록은 세션 종료 때 누적된 한 줄들을 흡수해 작성한다 (session.md 세션 종료 5단계 참조).

## 2. 막힘 탈출구 — Claude.ai 재질의

공장장이 막히면 오너를 찾기 전에 Claude.ai(설계 의도를 가장 잘 앎)에게 먼저 묻는다.

**트리거:** 같은 빌드 에러가 2회 이상 안 풀림 / 설계 방향 판단 필요 / 기존 코드와 충돌·해법 불분명 / 작업 완료 후 검수 요청

```bash
# 질문
node .claude/commands/ask-claude.js "질문: 내용"

# 완료 보고 (검수 요청)
node .claude/commands/ask-claude.js "완료보고: 작업명 / 변경파일 / 수용기준 충족여부"
```

**응답:** PASS → 다음 단계 / 수정 지시 → 재작업 후 재전송 / 오너 결정 필요 → 오너 에스컬레이션

스크립트: `.claude/commands/ask-claude.js`

### 오너 에스컬레이션 규약

Code는 오너에게 직접 에스컬레이션하지 않는다. 모든 오너 접촉은 ask-claude.js를 경유한다. 예외 없음 — 비즈니스 판단·개인 취향·인프라 이슈도 전부 경유. 경유 비용은 고정이지만, 경유 생략으로 생긴 불량은 비용이 폭발적이라 불량률 관점에서 경유가 경제적이다.

**경유 절차:**
1. Code가 ask-claude.js로 Claude.ai에 질의
2. Claude.ai가 답. 두 가지 경우:
   - "Code에게 지시 (오너 불필요)" — Code가 답 받고 공장 복귀
   - "오너 결정 필요" — Code가 Claude.ai 답변 + 원본 질문을 함께 오너에게 출력
3. Code 터미널 출력 시 헤더 표기: `[에스컬레이션: Claude.ai 경유]` + Claude.ai 분석 원문 + Code 원본 질문 + 오너 결정 필요 사유

**오너 반응:**
- `[에스컬레이션: Claude.ai 경유]` 헤더가 없는 메시지를 받으면 "ask-claude.js 경유로 다시 올려" 한 마디로 반려
- 반려 받은 Code는 즉시 ask-claude.js를 돌려 결과로 재출력

이 규약은 Claude.ai 웹 대화방과 Claude Code 터미널이 직접 통신할 수 없는 기술적 제약을 보완하는 장치다. Code가 "Claude.ai 경유했음"을 거짓 보고할 가능성도 있지만, Claude.ai 분석 문구의 구체성·설계 일관성으로 오너가 어느 정도 판별 가능. 의심되면 그 자체로 반려 사유.

## 3. 환경과 명령

**환경:**
- Claude Code (최신 버전)
- Codex: `openai/codex-plugin-cc` (`C:\Users\User\.claude\plugins\codex`)
- Playwright MCP: `C:\Users\User\.claude.json`
- GitHub Actions: `.github/workflows/ci.yml`
- 프로젝트: `D:\Dropbox\Dropbox\hizzi-board`
- 배포 URL: `https://hizzi-board.vercel.app`
- 세션 게이트: `scripts/preflight.ps1` (UserPromptSubmit hook으로 매 프롬프트 자동 실행, `.harness/session-started.flag` 존재 확인)

**Codex 슬래시 커맨드:**
```
/codex:review             — 일반 코드 리뷰 (read-only)
/codex:adversarial-review — 설계 결정·실패 모드 도전적 검열
/codex:rescue             — Claude Code 막혔을 때 Codex에 완전 위임
/codex:status             — 백그라운드 작업 상태
/codex:result             — 완료 작업 결과
/codex:cancel             — 진행 중 작업 취소
```

**빌드:**
```powershell
npm run build                                              # 기본
Remove-Item -Recurse -Force .next; npm run build           # 클린
git add . && git commit -m "..." && npx vercel --prod      # 배포
```
