# Hizzi Board — 기술 부채 관리

> 새 세션 시작 시 이 파일을 확인하고 우선순위 높은 것부터 처리 제안하세요.
> Claude는 새 기능 개발 전 이 파일을 먼저 체크하고 처리 여부를 오너에게 제안합니다.

---

## 처리 원칙

```
새 세션 시작 시    → 이 파일 확인 후 우선순위 높은 부채 처리 먼저 제안
새 기능 추가 전    → 관련 부채가 있으면 먼저 처리하는 게 효율적인지 판단
부채 처리          → 별도 세션으로 잡기 (기능 개발과 섞지 않기)
처리 완료 시       → ✅ 표시 + 완료 날짜 기록
```

## 🔴 우선순위 1 — 안정성 (곧 처리)

### 1. Panel.tsx 분리
```
현황: 615줄+ — 너무 많은 역할이 한 파일에 집중
위험: 버그 원인 특정 어려움 / 수정 시 사이드이펙트 위험

분리 계획:
  Panel.tsx         → 뼈대만 (탭, 패널 이름, 레이아웃 조합)
  TodoList.tsx      → 활성 할일 목록 전담
  CompletedTodo.tsx → 완료 할일 + 선택삭제 + 날짜 그룹핑
  PostList.tsx      → 메모/기타 게시물 목록 전담

작업 예상 규모: 파일 분리 + import 연결 → 빌드 검증 1회
관련 파일: Panel.tsx → TodoItem.tsx, PostItem.tsx, CreatePost.tsx
```

### 2. 에러 처리 통일
```
현황: 대부분 catch(e) { console.error(e) } → 사용자에게 피드백 없음
위험: 저장 실패해도 사용자가 모름 → 데이터 손실 가능성

개선 계획:
  모든 try-catch → addToast('오류 메시지') 추가
  Firestore 저장 실패 → "저장에 실패했습니다. 다시 시도해주세요."
  Firebase Storage 실패 → "파일 업로드에 실패했습니다."

해당 파일: TodoItem.tsx, CreatePost.tsx, Calendar.tsx,
           LeaveManager.tsx, todoRequestStore.ts
```

### 3. any 타입 제거
```
현황: any 타입이 곳곳에 남아있음
위험: 런타임에서 오류 발견 → 디버깅 어려움

주요 위치:
  TodoItem.tsx      → const updates: any = {}
  CreatePost.tsx    → postData: any
  Calendar.tsx      → deleteConfirm target: any, ev: any
  Panel.tsx         → CompletedRow p: any
  todoRequestStore  → docData: any (불가피한 경우 — 검토 필요)

개선 방향:
  PostUpdates 인터페이스 정의
  CalendarEventUpdates 인터페이스 정의
  LeaveEventData 인터페이스 정의
  Firestore 저장용 타입은 Partial<T> 활용
```

## 🟡 우선순위 2 — 성장 대비 (나중에)

### 4. Firestore 저장 공통 함수화
```
현황: undefined 제거 로직이 각 함수마다 따로 작성됨
위험: 한 곳에서 빠지면 Firestore 저장 실패

개선 계획:
  // src/lib/firestore.ts
  export function cleanFirestoreData<T extends object>(data: T): Partial<T> {
    const cleaned = { ...data };
    Object.keys(cleaned).forEach(key => {
      if ((cleaned as any)[key] === undefined) delete (cleaned as any)[key];
    });
    return cleaned;
  }

해당 파일: todoRequestStore.ts, postStore.ts, leaveStore.ts
```

### 5. 실시간 리스너 정리 검증
```
현황: initRequestListener 등 구독이 언마운트 시 정리되는지 불확실
위험: 메모리 누수 / 불필요한 Firestore 읽기 비용

확인 필요:

  Calendar의 onSnapshot이 cleanup 되는지
```

### 6. 공통 훅 추가 적용
```
useFileUpload(panelId)
  → TodoItem.tsx와 CreatePost.tsx의 Firebase Storage 업로드 로직 중복
  → 업로드 진행률, 에러 처리까지 통합 가능

useIsMobile(breakpoint)
  → Calendar.tsx와 Panel.tsx에 window.innerWidth 체크 중복

useCanEdit(ownerEmail)
  → isOwner / isAdmin 조합 체크가 Panel.tsx 곳곳에 중복

useMultiSelect(items)
  → Panel.tsx 완료 할일 선택삭제 로직 → 재사용 가능
```
