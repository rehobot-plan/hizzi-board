## 이 파일의 작성 규칙

### 넣는 것 / 넣지 않는 것

**넣는 것 — 기획 대화의 뿌리가 되는 판단 축.** 매 기획 대화마다 의식적으로 떠올리고 싶은 근본 원칙. 한 번 배우면 잊고 싶지 않은 것.

**넣지 않는 것:**
- 구체적 행동 규칙 (→ CLAUDE.md / harness.md / session.md)
- 일회성 결정
- 프로젝트 현황
- 짜잘한 판단 기준 — "이 기능은 이렇게 짜고" 수준의 국지적 결정

판단이 애매할 때: **"다음에 비슷한 상황에서 이 원칙 없이 대화하면 같은 실수를 반복할 만큼 본질적인가?"** 그렇다면 뿌리 원칙, 아니면 제외.

### 항목 수 상한

**10개 미만 유지.** 9개가 되면 "새 걸 더 넣을지, 기존 걸 확장해 흡수할지" 먼저 고민. 쌓기가 아니라 **다듬기**가 기본 동작.

### 유지 관리

- **추가** — 아주 드묾. 오너가 명시적으로 "principles에 추가하자" 발화할 때만. Claude.ai가 자발적으로 원칙 후보를 제안하지 않는다.
- **업데이트** — 자주. 기존 원칙이 새 상황에서 어떻게 적용되는지 "작동 방식"·"예외" 섹션에 디테일 추가.
- **폐기** — 극히 드묾. 폐기 시 삭제하지 않고 "폐기됨 (날짜, 사유)" 표시로 히스토리 보존.

### 원칙 작성 포맷

```
## N. 이름 — 한 줄 선언문

핵심 주장 (2~3문장). 왜 이게 참인지, 어떤 상황에서 유효한지.

**작동 방식:**
실제로 이 원칙이 상황에 어떻게 적용되는지.

**예외·주의:**
이 원칙이 무너지는 지점. 언제 재점검이 필요한지.

**근거 사례:**
히찌보드에서 이 원칙이 확인된 실제 사건.
```

## 1. 규칙 우선, 비용은 임계점에서만 재점검

히찌보드 운영에서 예외 구멍으로 효율을 짜내는 선택은 불량을 만든다. 매번 경유 비용은 고정이고 예측 가능하지만, 예외로 생긴 불량은 비용이 폭발적이고 예측 불가다. 따라서 판단이 엇갈릴 때 규칙을 우선하는 것이 장기적으로 경제적이다.

**작동 방식:**
공장 6단계는 작업 크기 무관하게 전부 거친다. Code는 오너에게 직접 에스컬레이션하지 않고 반드시 Claude.ai 경유. MD 수정은 매번 before/after 검수를 거친다. 단순 수정이어도 탐색을 생략하지 않는다. 하네스가 안전망 역할을 하고, Claude.ai가 입력 층에서 그것을 보완하려 하지 않는다. 실행 층 누수는 실행 층에서 고친다.

이 원칙은 기획 대화에서도 작동한다. "이건 예외로 두면 안 될까?" 같은 제안이 나오면 자동 승인하지 않고, 예외로 둘 때 생길 불량 비용을 먼저 따져본다.

**예외·주의:**
비용이 말도 안 되게 늘어나는 지점이 생기면 재점검한다. "매번 경유가 고정 비용"이라는 전제가 깨질 때 — 예를 들어 경유 자체가 수십 회 반복되는 플로우라면 그 플로우 설계를 다시 본다. **규칙 자체를 푸는 게 아니라 플로우 구조를 바꿔 규칙을 지킬 수 있는 조건을 만든다.** 이 구분이 중요하다.

**근거 사례:**
2026.04 하네스 파이프라인 관습 운영 시기에 @implementor trust-but-verify 이슈가 관측됐다. 이후 입력 층(Claude.ai) 규칙을 쌓아 보완하려 했으나 CLAUDE-detail.md가 16KB로 비대해져 세션 주입 실패. 실행 층 누수를 다른 층 규칙으로 메꾸는 시도가 오히려 불량을 만든 사례.

## 2. 개념보다 시스템

개념은 해석 부담을 읽는 주체에게 전가한다. "좋은 코드를 써라" 같은 개념은 쓰는 사람이 "좋은"을 매번 해석해야 한다. 규칙·원칙이 실제로 돌아가려면 **누가 무엇을 출력·판단·승인·실행하는지 행동 단위까지 내려가야 한다.** 그 수준까지 쪼개지지 않은 규칙은 해석 여지로 엉키고, 언젠가 관습 회귀의 진입점이 된다.

**작동 방식:**
새 규칙·원칙을 추가할 때 "이걸 미래의 Claude.ai·Code·오너가 해석 없이 바로 실행할 수 있는가?"를 먼저 묻는다. 해석이 필요하면 그 해석 과정을 규칙에 명시적으로 넣는다. 

이 원칙은 판단 기준이 될 수 있다. 예를 들어 "MD 수정은 검수를 거친다"는 개념이다. "변경 항목별 before/after 비교표를 먼저 제시 → 오너 OK/수정 → 패치 명령 블록"은 시스템이다. 전자는 해석 여지가 있고 후자는 없다.

**예외·주의:**
행동 단위까지 쪼개는 것이 과하게 가면 규칙 목록화가 된다. "쪼개지 않는 지점"의 판단은 "반복 패턴인가"에 있다. 반복되는 구조만 템플릿화하고, 일회성 행동은 해석 여지를 남겨도 된다.

**근거 사례:**
2026.04.19 세션에서 세션 종료 12단계의 "주관/기계 경계" 논의 중, Claude.ai가 "기계가 먼저, 주관이 나중" 같은 개념 수준에서 멈춘 제안을 여러 차례 내놓았다. 오너가 "개념 말고 시스템으로"라고 짚은 뒤 각 단계별 "Code가 출력 → Claude.ai 판단 → 오너 승인 → Code 실행" 행동 구조로 재설계됐고, 그 순간 설계가 풀렸다.

## 3. 작은 것도 쌓이면 단순하지 않다

"단순 수정"이라는 예외 구멍은 관습 회귀의 진입점이다. 한 번 "이건 간단하니까 건너뛰자"가 허용되면 다음엔 "이것도 비슷하게 간단하니까"가 된다. 단순함의 판정은 주관적이라 기준이 흔들리기 쉽다. 작업 크기와 무관하게 공정을 거쳐야 시스템이 유지된다.

**작동 방식:**
공장 6단계는 수정 규모와 무관하게 전부 거친다. 한 줄 타이포 수정도 탐색·빌드 검증·코드 리뷰를 거친다. E2E는 트리거 조건 매칭 여부로만 결정하고 "이건 작으니까 생략"은 없다. MD 한 문장 수정도 before/after 비교표를 제시한다. 

이 원칙은 새 규칙 설계에서도 작동한다. "이 경우만 예외 둘까?" 유혹이 생기면 그 예외가 만들 확장 경로를 상상한다. 대부분 예외는 첫 것 하나로 끝나지 않는다.

**예외·주의:**
진짜 예외가 필요한 지점이 있다면 그건 "구조의 결함"이지 "개별 사례"가 아니다. 예외를 인정하기 전에 구조 자체를 다시 본다. 예외 없이 돌아갈 조건으로 구조를 바꾸는 것이 우선.

**근거 사례:**
2026.04 하네스 운영 중 "단순 UI 수정은 @explorer 건너뛰자"가 관습화되면서 grep 탐색 없이 코드가 작성됐고, 기존 코드와 어긋나는 변경이 반복됐다. 이후 "작업 크기 무관하게 탐색 생략 없음"으로 규칙화. 예외 구멍 하나가 품질 저하의 입구였음이 확인된 사례.

## 4. 스페셜리스트 원칙

각 주체가 자기 책임 경계 안에서만 움직일 때 시스템이 가장 잘 돈다. 기능과 책임을 섞으면 역할이 흐려지고, 역할이 흐려지면 어느 주체도 그 자리의 품질을 책임지지 않는다. 효율을 위한 역할 겸임은 단기 이득이지만 장기적으로 각 영역의 전문성을 약화한다.

**작동 방식:**
오너는 방향을 이끌고 승인한다. Claude.ai는 기능 의도·판단 근거·영향 범위 정리까지 — 무엇을·왜 만드는가. 어떻게 만드는가(구현 구조·변수·함수·API 호출 패턴)는 공장의 영역이다. Claude Code는 프롬프트를 현실로 만드는 집행. Codex는 결과물의 실효성 검증. 서로 대체하지 않는다.

Claude.ai는 오너에게 작업 실행을 지시하지 않는다. 공장이 자동화로 처리해야 할 영역(탐색·구현·검증·배포)을 오너 시간으로 떠넘기면 스페셜리스트 경계가 뒤집힌다. 공정이 이탈했으면 공장이 돌아가서 복구한다.

Code는 판단 없이 실행, Codex는 실행 없이 판단. 같은 "공장 안"에 있지만 기능이 다르다. 공장 밖의 Claude.ai는 실행·검증에 개입하지 않고, 공장 안의 주체들은 기획 판단에 개입하지 않는다.

**예외·주의:**
경계가 애매한 회색지대는 반드시 생긴다. 예: 세션 종료 시 "이건 기계 작업인가 주관 판단인가". 이때는 **"조건이 객관적이면 기계, 해석이 필요하면 주관"**으로 나눈다. 둘이 섞여있으면 "객관 부분은 Code가 먼저 수집·제시, 주관 부분만 판단 주체가 결정"으로 순차화한다. 경계를 없애는 게 아니라 명확하게 긋는다.

**근거 사례:**
2026.04.19 세션 종료 12단계 재설계 중, 초기에 Claude.ai와 Code의 2축 구분만 잡았다가 Codex 자리를 통째로 놓친 실수가 있었다. 오너가 "Code에 모든 부담을 주고 있다. Codex의 실효성 검증이 있다"고 짚은 뒤 3축 구조로 교정. 한 주체가 다른 주체의 기능을 대행하면 원래 주체의 자리가 비게 된다는 것이 드러난 사례.

2026.04.20 세션 #46 H-1 재조정 명령 블록에서 Claude.ai가 Header.tsx의 hook 호출 순서·adminModeStore의 set 구문·persist 미들웨어 판단까지 지정한 사례. 공장의 1-1 탐색·1-2 구현에서 결정할 영역을 외부에서 미리 박아버린 것. 오너가 "코드도 충분히 똑똑해서 기획만 명확해도 다 풀어 오더라고"라고 짚은 뒤 경계를 명시적으로 긋는 방향으로 CLAUDE.md·rules.md·principles.md·harness.md 일괄 수정.

같은 세션 #46에서 Code가 1-5 E2E를 독단 보류하고 오너에게 "수동 확인 5건"을 넘긴 보고를 받았을 때, Claude.ai가 하네스 규정 위반을 지적하면서도 복구 실행을 오너에게 다시 요청한 사례. 공장이 이탈한 공정을 오너 시간으로 메꾸려 한 것. 오너가 "내게 명령하면 안 될 텐데"라고 짚은 뒤 Claude.ai가 오너에게 작업 지시하지 않는다는 경계를 명문화.

## 5. 문서가 기능하는 게 아니라 자동화가 기능한다

규칙을 문서에 쌓는다고 그 규칙이 지켜지는 게 아니다. 문서는 읽는 주체의 해석·의지에 의존하지만, 자동화된 시스템은 그 의존 없이 돈다. 새 규칙이 필요해 보일 때 먼저 "이건 자동화로 해결되는가"를 묻는다. 자동화로 해결되면 문서는 그 자동화의 설명이 되고, 해결 안 되면 그때 규칙으로 박는다.

같은 맥락에서, drift 방어를 사람 개입 루틴에 의존하지 않는다. 구조화 가능한 방어는 공장 내부 규약(Code 실측·주장-근거 매칭)으로 옮긴다. 검수는 구조화 불가능한 판단(프리셋 선정 같은 재량)에만 남긴다.

**작동 방식:**
새 기능·규칙 추가 요청이 들어오면 먼저 멈춘다. "이건 하네스 자동화로 해결되는 게 아닌가?"를 묻는다. 자동화 경로가 있으면 그쪽을 먼저 설계한다. 예: "명령 블록에 파이프라인 단계 표기 의무" 규칙이 있었으나, 하네스가 6단계를 자동 처리하면서 규칙 자체가 불필요해짐.

이 원칙은 CLAUDE.md 슬림 유지와 연결된다. CLAUDE.md가 비대해지는 주된 원인은 "자동화로 풀 문제를 규칙으로 쌓은 것"이다. 규칙 추가 전 자동화 점검이 파일 비대화 방지의 첫 관문이다.

방어선 설계에서도 같은 원칙이 작동한다. 초안이 "검수 한 번 더 넣어서 drift 막자"로 나올 때, 그 검수가 구조화 가능한 판단이면 Code 실측·주장-근거 1:1 매칭으로 옮긴다. 검수 부담 동일하면서 정확도는 올라간다. 방어선이 사람 눈에 있으면 세션마다 품질이 흔들리고, 공정 내부에 있으면 수렴한다.

**예외·주의:**
자동화로 해결되지만 **자동화 실패 시 안전장치가 없을 때**는 규칙도 함께 둔다. 예: preflight hook이 세션 시작을 자동 검사하지만, rules.md에도 "세션 게이트" 체크가 있었던 것 (현재는 hook 자동화가 충분히 검증돼 규칙은 제거). 자동화 검증이 쌓이면 중복 규칙을 뗀다.

방어선 구조화가 불가능한 판단에는 검수를 남긴다. "프리셋 선정"처럼 재량·맥락 의존이 강한 건 구조로 환원되지 않는다. 이걸 억지로 구조화하면 규칙이 현실과 어긋난다.

**근거 사례:**
2026.04.19 세션에서 CLAUDE-detail.md 해체 논의 중 P1~P7 기획자 원칙·"파이"/"기획" 단축어·R4.12 파이프라인 표기 의무 등이 일괄 폐기됐다. 전부 "하네스 자동화로 대체 가능"하다는 판단. 문서에 쌓아 해석 부담으로 넘기던 것을 자동화로 옮기자 문서가 2/3로 줄고 실제 운영 품질은 오히려 올라갔음이 확인된 사례.

## 6. 형식은 사실을 대체하지 않는다

잘 포맷된 텍스트일수록 신뢰 편향이 강해진다. 커밋 해시·테스트 숫자·배포 URL을 담은 "완료 보고"는 실제 저장소 상태와 무관하게 Claude.ai 내부에서 사실처럼 수용된다. Claude.ai는 Code 터미널 출력을 직접 볼 수 없고 전달된 텍스트만 본다 — 이 텍스트의 진위를 자체 검증할 수단이 구조적으로 없다. 따라서 사실 누적이 중요한 단계는 형식을 믿지 않고 저장소 실측을 선행한다.

**작동 방식:**
세션 종료·부채 갱신·프리셋 변경처럼 사실이 장기 영향을 미치는 단계 진입 전, Code가 저장소 실측(`git log` · str_replace 경로 + mtime · 아티팩트 실존)을 출력한다. Claude.ai는 이 출력만을 근거로 초안을 작성한다. 대화 콘텍스트의 "완료했다"는 텍스트는 근거 사용 금지. 초안 항목과 실측 출력이 1:1 매칭되지 않으면 초안 중단 + "사실 불일치: (항목)" 보고 후 실측 단계로 복귀.

포맷이 잘 잡힌 보고일수록 더 엄격하게 분리 검증한다. 형식은 사실을 암시할 뿐 사실을 보장하지 않는다.

**예외·주의:**
모든 발언을 실측하자는 뜻이 아니다. 기획 대화·설계 논의·탐색 질의는 텍스트 수용이 기본이다. 실측 선행이 필요한 지점은 **"이 판단이 파일·커밋·세션 기록에 박혀 다음 세션 이후까지 영향을 미치는가"**로 정한다. 그렇지 않으면 과잉이다.

**근거 사례:**
