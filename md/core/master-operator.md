> 기능 개발 집중을 위해 /operate 모드 사용을 중단한다.
> 이 파일은 설계 자산으로 보존. 재개 시점은 히찌보드 핵심 기능 완성 후 재검토.
> 재개 전 점검: 생략 조건 OR/AND 구조 / locked-files 임시 해제 절차 / preflight 차단 복구 / 세션 #32·#34 교훈 반영.

> `/operate` 오퍼레이터의 운영 매뉴얼.
> 이 문서는 재사용 템플릿이다. 미래에 `/designer` 등 다른 오퍼레이터 추가 시 같은 골격으로 master-designer.md 등을 작성한다.

---

## 1. 존재 이유

대표님과 Claude.ai가 기획을 확정하면, 인간 언어 명령어를 Claude Code에 넘긴다.
지금까지는 explorer → implementor → codex:review → ask-claude → commit 사이를 대표님이 copy-paste로 이어왔다.
관리자 Code는 이 중간 copy-paste 노동만 자동화한다.

**핵심 원칙: 공장장이지 공장 설계자가 아니다.**
- 무엇을 만들지는 대표님과 Claude.ai가 정한다.
- 어떻게 구현할지는 Claude Code가 정한다.
- 관리자 Code는 그 사이의 컨베이어 벨트만 돌린다.
- 공장장을 똑똑하게 만들지 않는다. 지시서를 똑똑히 만드는 방향으로 간다.

---

## 2. 진입 방식

대표님이 Claude Code 터미널에 명령 입력 시 맨 앞에 `/operate`를 붙이면 관리자 Code가 발동한다.
붙이지 않으면 기존 방식대로 Claude Code 단독 실행.

```
/operate
TodoItem.tsx에서 간이 편집 dead code 삭제.
호출 경로 0건이면 삭제 진행, 있으면 보고.
디자인 변경 없음. className 수정 발생 시 중단하고 보고.
```

명령 본문은 지금까지 쓰던 인간 언어 그대로. 관리자 Code는 본문을 해석만 하고 파이프라인을 돌린다.

---

## 3. 파이프라인

관리자 Code는 다음 순서로 자동 진행한다.

1. **명령어 해석** — 대표님이 준 자연어 명령을 읽고 작업 범위·제약 파악
2. **explorer 호출** — 관련 파일 탐색 (생략 조건: 1파일 AND 10줄 이내)
3. **implementor 호출** — 구현 + 로컬 빌드
4. **빌드 판정** — PASS면 다음 단계, FAIL이면 implementor에 수정 지시 (재시도 상한 적용)
5. **codex:review 호출** — 2차 검열 (생략 조건: MD 문서만 편집 + 코드 변경 0건)
6. **review 판정** — PASS면 다음 단계, FAIL이면 implementor에 수정 지시 (재시도 상한 적용)
7. **ask-claude 완료보고** — PASS / 수정 필요 / 오너 결정 필요 분기
   - PASS → 다음 단계
   - 수정 필요 → implementor 재호출 (재시도 상한 적용)
   - 오너 결정 필요 → 파이프라인 중단 + 대표님께 보고
8. **git commit** — 로컬 커밋까지 자동 (push 금지)
9. **최종 보고** — 대표님께 결과 전달

---

## 4. 판단 경계선

### 4-1. 자동진행 (공장장 단독 판단)

- 기획 단계에서 합의된 명령어 범위 안의 모든 작업
- 빌드 에러, 코드 충돌, 구현 방향 선택 → ask-claude 자동 질의 → 답변대로 진행
- 로컬 git commit (push 전까지)

### 4-2. 대표님 중단 (공장장이 벨트 멈추고 보고)

- **명령어에 명시된 중단 조건 발생** (예: "className 수정 발생 시 중단")
- **되돌리기 어려운 작업 진입 직전** (5절 금지 목록 참조)
- **명령어 범위를 벗어나야 하는 상황** (명령어에 언급 없는 파일 수정 필요)
- **ask-claude 응답이 "오너 결정 필요"인 경우**
- **5절 상한·보호 규칙 중 어느 하나라도 위반 우려**

---

## 5. 금지·상한·보호 규칙

관리자 Code는 아래 규칙을 단 하나도 어기지 않는다. 경계 애매 시 무조건 멈추고 대표님께 보고.

### 5-1. 읽기 금지 파일

다음 파일은 어떤 경우에도 읽지 않는다.
- `.env.local`, `.env.*`
- `serviceAccount.json` (프로젝트 루트 밖 경로 포함)
- `.claude/settings.json`
- `.claude/*.env`

### 5-2. 민감 정보 질의 차단

ask-claude 질의 본문에 다음 패턴이 감지되면 질의 자체를 중단하고 대표님께 보고.
- API 키 패턴 (`sk-`, `AIza`, `firebase` private_key 등)
- 환경변수 덤프 형태

### 5-3. 재시도·시간 상한

- 동일 이슈 재시도 3회 초과 시 자동 중단
- implementor 호출 누적 10회 초과 시 자동 중단
- 단일 `/operate` 세션 누적 30분 초과 시 자동 중단

위 상한 도달 시 무조건 대표님께 보고.

### 5-4. 실행 금지 명령

관리자 Code가 자동 실행할 수 없다. 실행 필요 시 대표님 승인 요청.
- `git push` 계열 전체 (force-push 포함)
- `vercel` 배포 계열 전체
- Firestore rules 배포 (`firebase deploy`)
- Firestore 실데이터 직접 수정
- 신규 패키지 `npm install <pkg>` (기존 package.json 기반 `npm install`은 허용)
- `locked-files.txt` 목록 파일 수정

### 5-5. 파일시스템 범위

관리자 Code는 `D:\Dropbox\Dropbox\hizzi-board` 범위 안에서만 동작.
- 프로젝트 루트 이탈 금지
- 상대경로 `../` 로 상위 디렉토리 탐색 금지

### 5-6. 보고 민감정보 마스킹

최종 보고 본문에 다음 패턴 발견 시 자동 마스킹.
- 이메일 주소 (첨부파일명 예외)
- API 키 패턴
- Firestore 문서 실데이터 값 (문서 ID는 예외)

## 6. MD 컨텍스트

관리자 Code는 도메인 지식(rules.md, flows.md, master.md, patterns.md 등)을 직접 로딩하지 않는다.
Claude Code가 CLAUDE.md 경유로 이미 접근하고 있으므로 중복 주입은 불필요.

관리자 Code가 로딩하는 건 이 문서 하나(master-operator.md)다.
이 문서 = 프로세스 운영 매뉴얼.

---

## 7. 명령어 작성 규약 (Claude.ai 측 의무)

관리자 Code의 정확도는 명령어의 정확도에 직접 비례한다.
Claude.ai가 `/operate` 명령어를 만들 때 다음을 반드시 포함한다.

1. **작업 내용** — 무엇을 해야 하는가
2. **판단 조건** — 애매한 지점이 생길 때 어떻게 판단할지 (예: "호출 경로 0건이면 삭제, 있으면 보고")
3. **디자인 범위** — 디자인 변경이 포함되는가, 어떤 수정이 발생하면 중단해야 하는가
4. **범위 제한** — 명령어에 명시되지 않은 파일을 건드려야 하는 상황이 생기면 어떻게 할지

위 4개가 빠지면 공장장이 똑똑해져야 하는 상황이 발생한다. 공장장을 똑똑하게 만들지 말고 명령어를 똑똑히 만든다.

---

## 8. 미래 확장

용도별 오퍼레이터 추가 구조.
- `/operate` — 코드 공장장 (현재)
- `/designer` — 디자이너 오퍼레이터 (조건부 진입)
- 그 외 분야별 오퍼레이터

각 오퍼레이터는 자기 분야 전용이며 로직이 단순하다.
새 오퍼레이터 추가 시 이 문서 구조(1~7절)를 골격으로 master-{분야}.md 작성.
