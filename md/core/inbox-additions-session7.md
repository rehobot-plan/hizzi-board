## [2026-04-10 세션 #7] 인박스 #1 갱신 — Codex 검열 강화 (우선순위 격상)

기존 인박스 #1에 검증 방법론 4가지 입력 추가. **다음 인프라 세션 최우선 처리**.

### 배경
세션 #7 4차 사이클에서 Opus가 수용 기준 8개 PASS 보고했으나 실제 렌더링은 4가지 결함 패턴으로 깨짐. 5차 사이클 보고서가 PASS 어긋난 4가지 원인 분석.

### 검증 방법론 결함 (5차 보고서 입력)
| 결함 | 대책 |
|---|---|
| 코드 라인 매칭으로 PASS 판정 | 금지. 반드시 실제 데이터로 실행 후 결과 확인 |
| 시드 데이터 부재 | Firestore 시드 데이터 스크립트 인프라 필요 |
| Codex review가 실행 경로 미추적 | Codex 프롬프트에 "이 입력이 이 코드를 통과할 때 실제 값이 무엇인지 trace" 요구 추가 |
| 스크린샷 검증 피상적 | 각 수용 기준별 expected vs actual row 번호 console.log 출력 + 비교 |

### 정책화 방향
- /codex:review 프롬프트에 "요구사항 충족 여부" + "실행 경로 trace" 항목 추가
- 명령 블록 템플릿에 "수용 기준(acceptance criteria)" 섹션 의무화
- 명령 블록에 "다중 케이스 시드 데이터 + 캡처 검증" 강제 조항
- 코드 검증만으로는 PASS 판정 금지

### 우선순위
**높음 (격상)** — 세션 #6/#7 연속 동일 패턴, 5차 사이클까지 누적

## [2026-04-10 세션 #7] 인박스 #3 신설 — 코드 스터디 정책

### 배경
세션 #7 5차 사이클까지 진행 후 오너가 발견: Opus가 외부 라이브러리 조사 0건. Claude.ai와 Claude Code 모두 "주어진 코드 안에서 답을 찾는다"는 자체 구현 프레임에 갇힘. FullCalendar 등 오픈소스가 이미 동일 문제 해결 (eventOrder, eventOrderStrict, compactness 자동 보정).

### 정책 요지
새 기능이나 큰 버그 수정은 자체 구현 시작 전 기존 라이브러리/오픈소스 조사를 1단계로 강제한다.

### 적용 4곳 (책임 분담)

| rules.md 마스터 체크리스트 | "기존 라이브러리/오픈소스 검토했는가?" 1줄 추가 (세션 게이트 아래) | 코드 작성 직전 |
| rules-detail.md R4.9 신설 | 코드 스터디 원칙 본문 (적용 대상/제외/매트릭스/사례) | 참조용 |
| CLAUDE.md Claude.ai 역할 | "외부 라이브러리 조사를 1단계로 제안 (R4.9)" 1줄 | 설계 시점 |

## R4.9 코드 스터디 원칙

새 기능이나 큰 버그 수정은 자체 구현 시작 전 기존 솔루션 조사를 1단계로 둔다.

### 적용 대상
- "범용 컴포넌트" 영역: 캘린더, 리치 에디터, 차트, 드래그앤드롭, 폼 빌더, 테이블, 다이어그램
- 자체 구현으로 3회 이상 시도해 미해결인 버그
- 추정 작업량이 1세션 이상인 신규 기능

### 적용 제외
- 히찌보드/Rehobot 비즈니스 로직 (게시판 카테고리, 결재 흐름, 권한 모델 등)
- UI 미세 조정 (색상, 간격, 애니메이션)
- 기존 컴포넌트의 작은 버그 수정

### 조사 단계 산출물
- 후보 라이브러리 2~3개 (이름, 라이센스, 번들 사이즈, 활성도, 한국어 처리)
- 우리 데이터 모델과의 호환성
- 자체 구현 vs 라이브러리 채택의 장단점
- 추천안 + 근거

### 의사결정 매트릭스
| 조건 | 권장 |
|---|---|
| 후보 라이브러리가 요구사항 90% 이상 커버 + MIT/Apache | 라이브러리 채택 |
| 후보가 60~90% 커버 | 라이브러리 + 어댑터 또는 fork |
| 60% 미만 또는 라이센스 충돌 | 자체 구현, 단 알고리즘은 참고 |
| 비즈니스 로직 영역 | 자체 구현 |
